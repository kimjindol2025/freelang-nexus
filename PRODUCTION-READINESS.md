## 📋 프로덕션 채택 체크리스트

### 1️⃣ **언어 수 스케일링** — 6개 → 10개

#### 현황
```
✅ 6개 언어 통과: Rust, Go, C, Zig, Julia, Python
❓ 10개 언어 미검증
```

#### 필요
- Mojo, V, Node.js, Kotlin 등 4개 언어 추가
- 각각 완전히 독립적인 FFI 테스트
- 복잡한 의존성 (A→B→C→D chain)

#### 체크 기준
```
✅ 10언어 polyglot demo 정상 실행
✅ 의존성 정렬 알고리즘 검증 (순환 참조 감지)
✅ 각 언어별 링크 플래그 충돌 테스트
```

#### 예상 결과
- 현재: "6개 언어는 됨"
- 기대: "10개 언어도 안정적"

### 2️⃣ **빌드 재현성** — 같은 소스 = 같은 결과

#### 현황
```
❓ 동일 .fl 파일의 10회 연속 빌드
  ├─ 컴파일 순서 변함?
  ├─ 최종 바이너리 해시 다름?
  ├─ 성능 편차?
  └─ 오류 메시지 일관?
```

#### 필요
```bash
# 10번 빌드 → 10개 바이너리가 모두 같은 해시
$ for i in {1..10}; do
    npm run build
    md5sum ./main >> hashes.txt
  done

# 기대: 모든 해시가 동일
$ sort hashes.txt | uniq
# sha256: abc123def456... (1줄 = 모두 동일)
```

#### 체크 기준
```
✅ 동일 소스 × 10회 = 동일 바이너리 해시
✅ 타임스탬프 제거 (비결정성 제거)
✅ 캐시 무효화 → 재빌드 일관성
```

#### 예상 결과
- 현재: "대부분은 같지만, 가끔 다를 수 있음"
- 기대: "100% 일관적"

### 3️⃣ **에러 메시징 품질** — 문제 정확히 설명

#### 현황
```
❌ 에러가 뜨면:
   - "undefined reference to 'hash_value'"
   - 어느 파일? 어느 줄? 왜 안 됨?
   - 해결책은?
```

#### 필요
```bash
# 나쁜 예
$ npm run build
Error: linking failed

# 좋은 예
$ npm run build
Error: [Link] Failed to link main.c with Go library
  ├─ Missing symbol: 'hash_value' (from hash.go, line 45)
  ├─ Reason: @export directive missing or function not C-compatible
  ├─ Linked libraries: [-lrust, -lc]
  ├─ Not linked: go library libhash.so
  └─ Fix: Add '@export' above the function declaration
```

#### 체크 기준
```
✅ 에러 시 정확한 파일:줄번호 표시
✅ 왜 실패했는지 기술적 이유
✅ 해결책 제시 (링크, 문서 참조)
✅ 시각적 강조 (색상, 구분선)
```

#### 예상 결과
- 현재: "기술 에러만 뜸"
- 기대: "사용자가 1분 안에 해결 가능"

### 4️⃣ **실패 격리** — 한 언어 에러가 전체를 먹지 않음

#### 현황
```
❌ 시나리오: Go 코드에 문법 에러
   → 전체 파이프라인 중단
   → C, Rust, Python 컴파일 진행 안 함
```

#### 필요
```bash
$ npm run build
[Lexing] ✅ 완료
[Parsing] ✅ 완료
[Codegen] ✅ 완료
[Compile/Go] ❌ FAIL: syntax error at line 45
  → 나머지 언어는 계속 컴파일
[Compile/Rust] ✅ 완료 (30s)
[Compile/C] ✅ 완료 (2s)
[Compile/Zig] ⏭️ 스킵 (Go 의존)
[Link] 불가능 (Go 실패)

Summary:
  ✅ Rust ✅ C (Go 실패로 링크 불가)
```

#### 체크 기준
```
✅ 각 언어 독립적 컴파일 (병렬 가능)
✅ 실패 시 다른 언어는 진행
✅ 의존성 있는 언어만 스킵
✅ 최종 요약: 뭐가 안 됐는지 명확
```

#### 예상 결과
- 현재: "하나 터지면 전체 멈춤"
- 기대: "디버깅 시간 50% 단축"

### 5️⃣ **ABI/타입 안정성** — "타입 불일치" 자동 감지

#### 현황
```
❌ 시나리오:
   Rust: pub extern "C" fn calculate(x: i32) -> i64
   V:    let result = calculate(x)  // 반환형 32비트로 잘못 인식?
   
   → 런타임 크래시 (타입 불일치)
```

#### 필요
```bash
# 빌드 타임에 감지
$ npm run build
[Codegen/Type Check]
  ❌ Type mismatch detected:
     - Rust function 'calculate' returns i64 (long long)
     - V function expects i32 (int)
     - Line main.v:45: let result = calculate(x)
     - Fix: Change to 'let result: i64 = calculate(x)'
```

#### 체크 기준
```
✅ Rust int ↔ V i32 자동 매핑
✅ 불일치 시 빌드 타임 에러
✅ C 타입 시스템 존중 (char*, void*)
✅ 포인터 안정성 (메모리 누수 감지)
```

#### 예상 결과
- 현재: "런타임에 터질 가능성"
- 기대: "컴파일 타임에 100% 검증"

### 6️⃣ **플랫폼 일관성** — Linux/Mac/Windows에서 동작

#### 현황
```
✅ Linux   (테스트됨)
❓ macOS   (문제 가능: dylib vs so)
❓ Windows (완전 미테스트)
```

#### 필요
```bash
# macOS에서
$ npm run build
[Compile/Rust] ✅ (dylib으로 컴파일, 자동 변환)
[Compile/Go]   ✅ (dylib으로 컴파일)
[Link]         ✅ (DYLD_LIBRARY_PATH 자동 설정)

# Windows에서
$ npm run build
[Compile/Rust] ✅ (dll로 컴파일, 자동 변환)
[Compile/Go]   ✅ (dll로 컴파일)
[Link]         ✅ (PATH 자동 설정)
```

#### 체크 기준
```
✅ Linux/macOS/Windows CI 테스트
✅ 플랫폼별 .so/.dylib/.dll 자동 선택
✅ 라이브러리 경로 자동 설정 (Linux: rpath, Mac: DYLD, Win: PATH)
✅ 크로스 컴파일 지원 (ARM, x86_64 등)
```

#### 예상 결과
- 현재: "Linux만 검증"
- 기대: "모든 플랫폼에서 같은 경험"

```markdown
| 메트릭 | 기존 | Nexus | 개선 |

| 컴파일 오류 디버깅 | 2h | 10m | 92% ↓ |
```

#### 체크 기준
```
✅ 정량적 비교: 라인 수, 시간, 파일 개수
✅ 정성적 비교: "더 이상 X를 걱정 안 함"
✅ 팀 피드백: 개발자 만족도
```

#### 예상 결과
