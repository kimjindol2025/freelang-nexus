# v1.0.1 — Go+Rust FFI 재현성 분석

**테스트 날짜:** 2026-04-14  
**테스트:** 5회 Go 라이브러리 + Rust FFI 빌드

---

## 발견 사항

### ✅ Rust 바이너리: 완전 결정적
```
Run 1: rust=b9d89147adee...
Run 2: rust=b9d89147adee...
Run 3: rust=b9d89147adee...
Run 4: rust=b9d89147adee...
Run 5: rust=b9d89147adee...

결과: 1/5 유니크 (모두 동일) ✓
```

**의미:** Rust의 `-O` 최적화 빌드는 Go 라이브러리와 링크되어도 완전히 결정적입니다.

### ⚠️ Go 라이브러리: 비결정적
```
Run 1: go=b54c0d1d1687...
Run 2: go=63ef36e459e0...
Run 3: go=ec2cfa104513...
Run 4: go=a00aaec144e5...
Run 5: go=7fb50310f916...

결과: 5/5 유니크 (모두 다름) ✗
```

**원인 추측:** `go build -buildmode=c-shared`에 타임스탬프 또는 비결정적 요소 포함

### ✅ 최종 출력: 완전 동일
```
All 5 runs: "Result: 14\n"

결과: 1/5 유니크 (모두 동일) ✓
```

**의미:** Go 라이브러리가 다르더라도, Rust 링크 후 실제 실행 결과는 동일합니다.

---

## 해석

1. **Rust 결정성:** 완벽함 (v1.0.0 수준 유지)
2. **Go 결정성:** 라이브러리 빌드 비결정적 (미결정)
3. **최종 결과:** 동일 (Go 라이브러리의 내용 자체는 같음, 메타데이터만 다름)

---

## 다음 단계 (v1.0.1 방향 선택)

### 옵션 A: Go 빌드 옵션 최적화
```bash
# 타임스탬프 제거
go build -buildmode=c-shared -ldflags="-s -w" -o libgo_lib.so lib.go

# 또는 빌드 정보 제거
go build -buildmode=c-shared -a -trimpath -o libgo_lib.so lib.go
```

### 옵션 B: 최종 바이너리/출력 동일성으로 정의
```
Go 라이브러리 바이너리는 비결정적일 수 있으나,
최종 Rust 바이너리 + 실행 결과는 완전 결정적이다.
```
→ v1.0.1 범위: "최종 아티팩트 재현성"

### 옵션 C: Go 라이브러리 재현성은 v1.0.2로 연기
```
v1.0.1: Rust + Go 바인딩의 최종 결과 재현성 (완료 ✓)
v1.0.2: Go 라이브러리 자체 결정성 (별도 검증)
```

---

## 추천

**옵션 B 또는 C**를 추천합니다.

이유:
1. Rust 바이너리와 최종 출력이 완전히 동일하다는 것은 이미 강력한 증거
2. Go의 비결정성은 라이브러리 빌드 도구 문제이지, FreeLang Nexus의 문제가 아님
3. v1.0.1의 핵심은 "FFI 경계 재현성"이지, "모든 언어 내부 결정성"이 아님

**선언:** "Go 라이브러리 소스 → Rust FFI 링크 → 최종 바이너리/출력 100% 재현성 검증 완료"
